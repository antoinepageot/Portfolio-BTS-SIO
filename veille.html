Présentation du sujet

Ma veille technologique porte sur le modèle Zero Trust Security, une approche moderne de cybersécurité formalisée dans la publication NIST Special Publication 800-207. Ce modèle part d'un constat simple : les périmètres réseau traditionnels (pare-feu, VPN) ne suffisent plus face aux environnements hybrides cloud/on-premise et à la généralisation du télétravail.

Pourquoi ce sujet ?

  • Développement massif du télétravail et des environnements cloud
  • Augmentation des attaques par ransomware ciblant les AD et les identités
  • Adoption par Microsoft, Google, Cisco comme standard de référence
  • Lien direct avec mes missions E5/E6 : AD, GPO, permissions NTFS, MFA

Fiche #01

Publication NIST SP 800-207 — Formalisation du modèle Zero Trust Architecture

Moyen
Date de veille 14/10/2024
Source NIST – SP 800-207
Thème Architecture Zero Trust
Résumé

Le NIST a publié la SP 800-207, référence officielle du modèle Zero Trust Architecture. Elle définit 7 principes fondamentaux : vérification systématique des identités, accès au moindre privilège, inspection de tous les flux, micro-segmentation, refus implicite par défaut, et authentification continue. Trois approches d'implémentation sont distinguées : basée sur l'identité (IAM), sur le réseau (micro-segmentation), ou sur les données.

Impact

Toutes les entreprises utilisant un modèle de confiance implicite basé sur le périmètre réseau traditionnel. Ce standard redéfinit la manière d'administrer les accès au sein d'un Active Directory et des politiques GPO.

Action recommandée

Revoir la segmentation des accès dans l'AD (groupes, OU, délégation), renforcer les politiques MFA, documenter tous les flux entre ressources. Conserver la SP 800-207 comme référence pour tout projet d'infrastructure sécurisée.

Lien avec ma formation

Directement lié à ma mission E5 & E6 chez IPACS : la création des OU, groupes, GPO et la gestion des permissions NTFS s'inscrit dans la logique du principe du moindre privilège défini par NIST 800-207.

Fiche #02

Microsoft Entra ID – Évolution d'Azure AD vers une plateforme d'identité Zero Trust

Élevé
Date de veille 06/03/2025
Source Microsoft Learn – Entra ID
Thème Identité & Zero Trust
Résumé

Microsoft a renommé Azure Active Directory en Microsoft Entra ID et continue d'enrichir la plateforme pour en faire le socle de sa stratégie Zero Trust. Entra ID intègre nativement le MFA, l'accès conditionnel (Conditional Access), la protection des identités (Identity Protection), la gestion des accès privilégiés (PIM) et le SSO. Elle permet de mettre en œuvre le principe "ne jamais faire confiance, toujours vérifier" à l'échelle d'un domaine hybride.

Impact

Les entreprises utilisant Windows Server avec Active Directory et souhaitant étendre leur gestion des identités vers le cloud. L'accès conditionnel peut bloquer ou autoriser des connexions selon le niveau de risque ou le type de terminal. Les comptes administrateurs sont directement ciblés par PIM pour éviter les élévations permanentes de droits.

Action recommandée

Activer le MFA pour tous les comptes, en priorité les administrateurs. Mettre en place des politiques d'accès conditionnel pour les accès distants. Utiliser PIM pour que les élévations de droits soient temporaires et tracées.

Lien avec ma formation

Chez IPACS, l'infrastructure repose sur un AD on-premise (Windows Server 2025). Ce sujet m'a permis de comprendre comment faire évoluer cet environnement vers un modèle hybride. La création des groupes et OU dans ma mission E5 préfigure une future intégration Entra ID.

Fiche #03

CVE-2024-49113 – Faille critique Windows LDAP (CVSS 9.8) – Contrôleurs de domaine AD

Critique
Date de veille 18/11/2025
Source MSRC Microsoft · CERT-FR
Thème Sécurité AD / CVE
Résumé

Une vulnérabilité critique de type Remote Code Execution (RCE) a été découverte dans le service Windows LDAP, composant central de l'Active Directory. Référencée CVE-2024-49113 avec un score CVSS de 9.8, elle permet à un attaquant non authentifié d'exécuter du code arbitraire à distance sur un contrôleur de domaine via un paquet LDAP malformé. Les versions affectées couvrent Windows Server 2019, 2022 et 2025. Correctif publié dans le Patch Tuesday de décembre 2024.

Impact

Tous les contrôleurs de domaine exposant le port LDAP (389/TCP) sans filtrage réseau. Une compromission entraîne la prise de contrôle totale de l'Active Directory. Cette faille illustre pourquoi le modèle Zero Trust interdit de supposer qu'un réseau interne est sécurisé.

Action recommandée

Appliquer le correctif Microsoft KB5048667 (Patch Tuesday décembre 2024). Restreindre l'accès aux ports LDAP/LDAPS aux seules machines autorisées. Activer la journalisation des événements LDAP. Mettre en place une supervision des tentatives d'authentification anormales.

Lien avec ma formation

Cette faille concerne directement le Windows Server 2025 déployé chez IPACS dans ma mission E5. J'ai configuré l'Observateur d'événements pour surveiller les accès au domaine ipacs.local. Cette CVE renforce la pertinence du modèle Zero Trust étudié en fiche #01 : exposer un contrôleur de domaine sans micro-segmentation contredit les principes NIST 800-207.
